# 服务器探针产品需求文档 PRD

## 1. 背景

很多个人开发者和小团队会在一台或少量服务器上部署多个项目。日常运维中常见问题包括：

- 不知道服务器当前 CPU、内存、磁盘和负载是否健康。
- 不清楚某个项目跑在哪个端口、进程是否存活、启动方式是什么。
- 重启项目需要登录服务器手动执行命令，容易误操作。
- 查日志依赖 SSH 和命令行，对非熟练运维人员不友好。
- 发生问题后缺少审计记录，不知道谁在什么时候做过什么操作。
- 告警能力缺失，资源过高时通常只能事后发现。

本产品希望提供一个轻量、清晰、可控的服务器探针与运维面板，先解决单服务器项目运维，再扩展到多服务器和告警联动。

## 2. 产品目标

### 2.1 一句话定位

一个面向个人开发者和小团队的服务器状态监控、项目管理、日志查看和受控运维操作面板。

### 2.2 核心目标

- 降低服务器状态查看和项目运维门槛。
- 将项目启动、停止、重启等高频操作产品化。
- 让日志排查从命令行迁移到可检索、可筛选的界面。
- 对所有敏感操作留下审计记录。
- 为后续告警、多服务器 Agent 和自动化运维打基础。

### 2.3 非目标

第一阶段不做完整 APM、CI/CD、堡垒机或云管平台。

暂不追求：

- 分布式链路追踪。
- Kubernetes 集群管理。
- 完整发布流水线。
- 交互式 Web Shell。
- 大规模监控系统替代品。

### 2.4 产品形态

第一阶段是单服务器 Web 应用：前端 Web 和后端 API 部署在同一台服务器，后端 Nitro/h3 进程同时承担本机探针 Agent 的职责。

后续阶段再演进为独立 Agent + 中心管控架构。

## 3. 目标用户

### 3.1 个人开发者

有 1 到 3 台云服务器，部署了博客、API、管理后台、小工具等项目。希望不用频繁 SSH，也能知道服务是否正常。

### 3.2 小团队负责人

管理少量内部系统，希望团队成员可以查看状态和日志，但只有少数人能执行重启、停止等操作。

### 3.3 轻量运维人员

需要快速判断服务器资源、进程、端口和项目状态，并保留操作记录。

## 4. 使用场景

### 4.1 查看服务器是否健康

用户打开首页，看到 CPU、内存、磁盘、负载、网络、运行时间、系统版本等信息，能够快速判断服务器是否异常。

### 4.2 排查项目无法访问

用户进入项目列表，查看项目端口、进程状态、启动方式和部署路径。若项目异常，可以查看日志或执行重启。

### 4.3 查看实时日志

用户进入日志页面，选择项目，查看实时日志流，也可以按关键词搜索历史日志。

### 4.4 处理端口占用

用户进入进程管理，查看端口占用和相关进程，判断是否存在异常进程。

### 4.5 追溯操作记录

项目被重启后，管理员可以在审计记录中看到操作者、时间、目标项目、操作类型和操作结果。

### 4.6 资源过高告警

后续版本中，当 CPU、内存或磁盘持续超过阈值，系统通过邮件、企业微信、飞书或 Webhook 通知用户。

## 5. 功能范围

### 5.1 服务器状态

功能说明：

- 展示 CPU 使用率。
- 展示内存总量、已用、可用、使用率。
- 展示磁盘分区、容量、已用、可用、使用率。
- 展示系统负载。
- 展示网络上行、下行速率。
- 展示运行时间。
- 展示系统版本、内核版本、架构、主机名。

验收标准：

- 首页可以在 3 秒内展示当前服务器摘要。
- 指标刷新不阻塞页面交互。
- CPU、内存、磁盘异常时有明显状态标记。

优先级：P0

### 5.2 项目列表

功能说明：

- 查看当前纳管项目。
- 展示项目名称、端口、进程状态、启动方式、部署路径。
- 展示最近一次操作时间和操作结果。
- 支持按名称、端口、状态筛选。

验收标准：

- 用户可以快速找到某个项目。
- 项目状态至少包含运行中、已停止、异常、未知。
- 部署路径可以复制。

优先级：P0

### 5.3 项目操作

功能说明：

- 对纳管项目执行启动、停止、重启。
- 查看项目实时状态。
- 操作前弹出确认。
- 操作后展示结果、耗时、失败原因。
- 同一个项目同一时间只允许一个操作执行。

验收标准：

- 所有项目操作必须写入审计记录。
- 未登录或无权限用户不能执行操作。
- 命令执行超时后自动终止并返回错误。

优先级：P0

### 5.4 日志查看

功能说明：

- 按项目查看实时日志。
- 查看历史日志。
- 按项目筛选。
- 按关键词搜索。
- 支持日志级别识别：info、warn、error。
- 支持暂停实时滚动。
- 实时日志传输使用 SSE，接口为 `GET /api/logs/stream?projectId=`。

验收标准：

- 实时日志延迟应尽量低于 2 秒。
- SSE 连接断开后前端可以自动重连。
- 单次历史日志读取有行数和大小限制。
- 搜索高亮关键词。

优先级：P0

### 5.5 进程管理

功能说明：

- 查看当前进程列表。
- 查看端口占用。
- 标记 CPU 或内存占用异常的进程。
- 查看进程 PID、名称、CPU、内存、启动时间、命令摘要。

验收标准：

- 支持按 CPU、内存排序。
- 支持按端口查进程。
- 默认不提供 kill 进程能力。

优先级：P1

### 5.6 运维操作

功能说明：

- 清理缓存。
- 查看环境变量。
- 查看磁盘目录占用。

安全要求：

- 环境变量默认脱敏。
- 清理缓存必须基于白名单目录。
- 目录占用只能查看允许路径，避免遍历敏感目录。

验收标准：

- 用户可以看到每次清理的目标、释放空间和执行结果。
- 高风险操作必须二次确认。

优先级：P1

### 5.7 审计记录

功能说明：

- 记录谁在什么时候操作了哪个项目。
- 记录操作类型、目标、参数摘要、结果、错误信息、IP、User-Agent。
- 支持按用户、项目、操作类型、结果、时间筛选。

验收标准：

- 所有写操作必须落审计。
- 审计记录不可由普通用户删除。
- 审计列表支持分页。

优先级：P0

### 5.8 告警扩展

功能说明：

- 配置 CPU、内存、磁盘阈值。
- 配置持续时间，避免瞬时波动误报。
- 支持邮件、企业微信、飞书、Webhook。
- 支持告警恢复通知。

验收标准：

- 同一告警支持静默周期，避免刷屏。
- 告警触发、恢复、发送失败均有记录。

优先级：P2

## 6. 产品决策与补充需求

### 6.1 登录认证与角色权限

决策：登录认证与角色权限作为 P0 实现。

原因：

- 项目重启、停止、缓存清理都属于高风险操作。
- 即使是个人项目，也不应裸奔暴露在公网。

角色：

- 管理员：所有权限。
- 运维：查看、项目操作、日志、进程、运维工具。
- 只读：只可查看状态、项目、日志和审计。

### 6.2 项目纳管配置

决策：MVP 使用 `config/projects.yaml` 作为项目纳管配置源。

项目操作不能依赖页面上临时输入命令，必须来自配置文件中的固定动作。后台配置能力可以后续补充，但写入前必须经过同一套 schema 校验。

配置格式：

```yaml
version: 1
projects:
  - id: my-api
    name: My API
    deploy_path: /srv/my-api
    port: 3000
    process_manager: pm2
    start_cmd: "pm2 start ecosystem.config.cjs --only my-api"
    stop_cmd: "pm2 stop my-api"
    restart_cmd: "pm2 restart my-api"
    status_cmd: "pm2 describe my-api"
    log_files:
      - /var/log/my-api/app.log
    health_check:
      type: http
      url: http://127.0.0.1:3000/health
```

关键规则：

- `id` 必须全局唯一，审计记录和日志筛选都使用该字段关联项目。
- `deploy_path`、`log_files` 必须是绝对路径。
- `start_cmd`、`stop_cmd`、`restart_cmd`、`status_cmd` 只能由配置文件提供，前端不得传入任意命令。
- 配置加载失败时，项目操作能力必须禁用，只允许查看错误信息。
- 后续如迁移到数据库，仍需保留导入/导出 `projects.yaml` 的能力。

### 6.3 实时日志传输机制

决策：实时日志使用 SSE，不使用 WebSocket 作为 MVP 默认方案。

原因：

- 日志流是后端到前端的单向推送，SSE 更轻量。
- Nitro/h3 适合实现 SSE event stream。
- 浏览器原生 `EventSource` 支持断线重连。

接口：

- `GET /api/logs/stream?projectId=`

事件类型：

- `log`：日志行。
- `heartbeat`：心跳，保持连接。
- `error`：日志文件不可读、权限不足或项目不存在。

WebSocket 保留给后续需要双向交互的场景，不作为第一阶段日志方案。

### 6.4 数据存储策略

决策：MVP 使用文件配置 + SQLite 的组合。

存储边界：

- 项目配置：`config/projects.yaml`，作为项目纳管的事实来源。
- 用户、角色、会话：SQLite，默认文件为 `data/server-probe.sqlite`。
- 审计记录：SQLite，所有写操作必须落库。
- 指标实时数据：内存 ring buffer，优先保存最近 1 小时，用于总览页趋势展示。
- 指标历史数据：P1 阶段落 SQLite，默认保留最近 24 小时到 7 天，具体保留周期可配置。
- 告警规则与告警事件：P2 阶段落 SQLite。
- 日志内容：不复制入数据库，直接从 `projects.yaml` 中声明的日志文件读取和流式推送。

### 6.5 Agent 形态定义

决策：第一阶段的“探针 Agent”就是本机运行的 Nitro/h3 后端进程。

阶段定义：

- 第一阶段：前端 Web 和后端 API 部署在同一台服务器，后端进程负责本机指标采集、日志读取和受控项目操作。
- 后续阶段：支持独立 Agent + 中心管控分离。Agent 负责采集和执行，中心端负责用户、审计、告警和多服务器视图。

第一阶段不引入远程 Agent 注册、远程指令下发和多服务器数据隔离，但代码模块应预留 `agent`、`system`、`projects` 的边界。

### 6.6 部署方式

决策：MVP 支持两种部署方式，优先保证个人开发者 10 分钟内完成单服务器部署。

部署方式：

- 推荐方式：Docker Compose，暴露前端 `5100` 和后端 `5000`，挂载 `config/`、`data/` 和需要读取的日志目录。
- 备选方式：源码部署，使用 pnpm 构建后由 systemd 管理前端和后端进程。

暂缓方式：

- 单二进制分发不进入 MVP，可作为后续 P2 研究方向。
- npm/pnpm 全局安装不作为生产推荐方式，仅可用于开发调试。

### 6.7 指标历史趋势

决策：指标历史趋势作为 P1 实现。

仅看实时状态不够排查问题，至少保留最近 1 小时或 24 小时的 CPU、内存、磁盘、网络趋势。

### 6.8 备份与恢复提示

决策：备份与恢复提示作为 P2 实现。

不是做完整备份系统，而是在危险操作前提示：

- 最近部署时间。
- 最近备份时间。
- 是否存在未确认风险。

### 6.9 Docker / Docker Compose 项目纳管

决策：纳管 Docker / Docker Compose 项目作为 P2 实现。

很多个人服务器项目都用 Docker Compose 部署。后续可以支持：

- 查看容器状态。
- 查看容器日志。
- 重启服务。
- 查看镜像和端口映射。

### 6.10 安全暴露检测

决策：安全暴露检测作为 P1 实现。

可以检测一些常见风险：

- 是否监听 `0.0.0.0`。
- 是否存在未授权开放端口。
- 磁盘即将写满。
- 日志文件异常膨胀。
- 是否存在长期高占用进程。

## 7. 暂缓或不做范围

### 7.1 Web Shell

决策：MVP 和 V1 均不提供浏览器里的 SSH 或任意命令执行。

原因：

- 风险极高。
- 容易绕过审计。
- 权限边界难控制。
- 一旦暴露公网，攻击面很大。

替代方案是白名单运维操作。

### 7.2 复杂告警平台

决策：MVP 只预留告警模型，不实现完整告警平台。

原因：

- 告警涉及规则引擎、通知渠道、静默、恢复、失败重试。
- 如果前期指标和项目管理还没稳定，告警很容易反复返工。

完整告警能力放入第二阶段。

### 7.3 多服务器统一管理

决策：MVP 不做多服务器统一管理。

原因：

- 多服务器涉及 Agent 注册、鉴权、心跳、离线、升级、数据隔离。
- 会显著增加复杂度。

第一阶段只预留架构边界。

### 7.4 清理缓存

决策：清理缓存保留在 P1，但只允许操作白名单目录。

清理缓存容易误删文件。第一版只支持明确白名单目录，例如：

- 项目临时目录。
- 包管理缓存。
- 日志归档目录。

不提供“全盘扫描后一键清理”。

## 10. 关键业务规则

### 10.1 项目状态规则

- 运行中：端口监听存在，且进程匹配项目配置。
- 已停止：端口未监听，进程不存在。
- 异常：端口存在但进程不匹配，或进程存在但健康检查失败。
- 未知：检测命令失败或权限不足。

### 10.2 日志规则

- 日志文件路径必须来自项目配置。
- 单次读取默认最多 1000 行。
- 实时日志使用 SSE 和浏览器 `EventSource`。
- SSE 断线后允许重连。
- WebSocket 不作为 MVP 日志传输方案。
- 前端可以暂停自动滚动。

### 10.3 审计规则

必须记录的操作：

- 登录成功和登录失败。
- 项目启动。
- 项目停止。
- 项目重启。
- 缓存清理。
- 告警规则新增、修改、删除。
- 用户权限变更。

### 10.4 告警规则

告警触发应满足：

- 指标超过阈值。
- 持续时间达到配置要求。
- 当前不在静默周期。

告警恢复应满足：

- 指标回落到阈值以下。
- 持续稳定一段时间。

## 11. 风险与对策

| 风险 | 说明 | 对策 |
| --- | --- | --- |
| 命令执行风险 | 项目操作需要调用系统命令 | 使用白名单、超时、审计、固定工作目录 |
| 权限风险 | 暴露在公网后可能被攻击 | 登录认证、Token、HTTPS、限流 |
| 误操作风险 | 停止或重启错误项目 | 二次确认、显示目标项目、操作锁 |
| 日志过大 | 大文件读取拖垮后端 | 限制行数、限制大小、流式读取 |
| 跨平台差异 | Linux、Windows 命令不同 | 第一阶段明确优先 Linux，Windows 开发环境仅用于开发 |
| 数据丢失 | 审计和配置丢失 | SQLite 文件备份，后续支持 PostgreSQL |

## 12. 版本规划

## 13. 成功指标

- 用户 30 秒内判断服务器是否健康。
- 用户 1 分钟内找到某个项目状态和日志。
- 项目重启操作全程可追溯。
- 日志查看不需要 SSH。
- 关键写操作 100% 有审计记录。
- 单服务器部署流程少于 10 分钟。

## 14. 第一阶段结论

第一阶段聚焦：

- Nuxt + Nitro/h3 全栈骨架。
- 单服务器状态探针。
- 项目纳管。
- 受控项目操作。
- 实时日志。
- 审计记录。
- 登录认证。

不要一开始就做多服务器、Web Shell、复杂告警和自动化部署。这个项目最重要的是“可信”和“可控”：用户敢把它放到服务器上，敢用它重启项目，也能在出问题时查清楚发生了什么。
